操作系统的异步性导致本来需要有先后顺序的计算而无法保证先后，那么需要的值可能会被更新，所以需要加以约束，此时需要引入**同步的概念**

# 同步和互斥的基本概念

## 同步
**临界资源**：同一时刻仅允许一个进程访问的资源
既有硬件也有软件
例：打印机，共享变量，数据结构

**对临界资源必须要互斥访问**
**临界区**：访问临界资源的代码，不能用多个进程同时执行这段代码
有以下四个部分
1. 进入区：检查当前能不能进，进入后防止其他进程进入
2. 临界区：访问临界资源的代码段
3. 退出区：释放
4. 剩余区

**同步**
协调进程的运行次序，再执行过程产生等待或传递信息的制约关系

**互斥**
当一个进程正在临界区使用临界资源的时候，其他访问改资源的进程需等待，只有没有人用的时候才可以进入 

**同步机制：**

**空闲让进
忙则等待
有限等待
让权等待**

## 信号量

用来表示某种资源的数量，一个打印机S = 1
信号量需要使用两个标准的原语（不可被中断）
`wait(),signal()`
`P(S)操作  ，V(S)操作`

1. 整形信号量
用一个正数表示某种资源
init
P
V
```C
int S = 1;  // 初始化整型信号量s，表示当前系统中可用的打印机资源数

void wait (int S) {      //wait 原语，相当于“进入区”
    while (S <= 0);       //如果资源数不够，就一直循环等待
    S=S-1;               //如果资源数够，则占用一个资源
}

void signal (int S) {    //signal 原语，相当于“退出区”
    S=S+1;               //使用完资源后，在退出区释放资源
}

```

1. 记录型信号量
```C
/*记录型信号量的定义*/
typedef struct {
    int value;          //剩余资源数
    struct process *L;  //等待队列
} semaphore;

/*某进程需要使用资源时，通过 wait 原语申请*/
void wait (semaphore S) {
    S.value--;
    if (S.value < 0) {
        block (S.L);
    }
}

/*进程使用完资源后，通过 signal 原语释放*/
void signal (semaphore S) {
    s.value++;
    if (S.value <= 0) {
        wakeup(S.L);
    }
}

```

## 生产者消费者

---
单生产者消费者
![](Pasted%20image%2020260610202911.png)
```
两个同步信号量 + 缓冲区信号量
semaphore full
semaphore empty 
semaphore plact

互斥信号量
semaphore mutex
```

```c
semaphore empty = n;    // 缓冲区空位数（初始n个空位）
semaphore full = 0;     // 缓冲区产品数（初始0个产品）
semaphore mutex = 1;    // 互斥访问缓冲区

producer () {
    while (1) {
        生产一个产品;
        P(empty);        // 先申请空位，没空位则阻塞
        P(mutex);        // 互斥访问缓冲区
        把产品放入缓冲区;
        V(mutex);
        V(full);         // 产品数+1，唤醒消费者
    }
}

consumer () {
    while (1) {
        P(full);         // 等有产品才能取
        P(mutex);
        从缓冲区取出一个产品;
        V(mutex);
        V(empty);        // 空位数+1，唤醒生产者
        使用产品;
    }
}
```


- **同步关系**：缓冲区空 → 生产者可放入（`empty`）；缓冲区有产品 → 消费者可取（`full`）
- **互斥关系**：缓冲区同一时刻只能一个进程访问（`mutex`）
- 两个 `P` 操作不可颠倒（同理消费者 `P(full)` 先于 `P(mutex)`）

**能否在互斥信号量和同步信号量更换位置**
![](Pasted%20image%2020260610203221.png)
一旦生产者执行代码，更换后会直接占用缓冲区，但是此时已经满了，就在`P(empty)`的位置卡住，和消费者会产生死锁
> **关键**：`P(empty)` 必须在 `P(mutex)` 之前。若反过来，缓冲区满时生产者持 mutex 阻塞 → 消费者无法 P(mutex) 取产品 → **死锁**。

多生产者多消费者

有两组进程，一组专门产生资源，另一组专门消费产出的资源
他们共享一段缓冲区，缓冲区只能放一种资源
需要对缓冲区进行互斥访问
![](Pasted%20image%2020260610201501.png)
需要设置同步信号量和互斥信号量
```
两种资源：同步关系
semaphore apple
semaphore orange
semaphore plate 盘子还可以放几个苹果
互斥信号量
mutex 盘子

```c
semaphore apple = 0;    // 盘子中苹果的数量
semaphore orange = 0;   // 盘子中橘子的数量
semaphore plate = 1;    // 盘子还可以放几个水果（容量为1）
semaphore mutex = 1;    // 互斥访问盘子

dad () {
    while (1) {
        准备一个苹果;
        P(plate);        // 盘子有空位才能放
        P(mutex);        // 互斥访问盘子
        把苹果放入盘子;
        V(mutex);
        V(apple);        // 通知消费者：苹果+1
    }
}

mom () {
    while (1) {
        准备一个橘子;
        P(plate);
        P(mutex);
        把橘子放入盘子;
        V(mutex);
        V(orange);
    }
}

daughter () {
    while (1) {
        P(apple);        // 盘子里有苹果才能取
        P(mutex);
        从盘中取出苹果;
        V(mutex);
        V(plate);        // 盘子空位+1
        吃掉苹果;
    }
}

son () {
    while (1) {
        P(orange);
        P(mutex);
        从盘中取出橘子;
        V(mutex);
        V(plate);
        吃掉橘子;
    }
}
```

> **核心思路**：盘子容量为1 → `plate=1`。生产者先 `P(plate)` 占用空位，放入后 `V(apple/orange)` 通知对应的消费者。消费者先 `P(apple/orange)` 等待对应的水果，取出后 `V(plate)` 归还空位。`mutex` 保证同一时刻只有一人操作盘子。

- **同步关系**：盘子空 → 生产者可放入（`plate`）；水果就绪 → 对应消费者可取（`apple/orange`）
- **互斥关系**：所有进程对盘子的访问互斥（`mutex`）
- **为什么 P 操作顺序不能换**：先 `P(plate)` 再 `P(mutex)`，避免死锁。若反过来，生产者拿到 mutex 后发现盘子满 → 阻塞 → 消费者无法拿到 mutex 取水果 → 死锁
## 读写
 1. 允许多个读者同时读
 2. 只允许一个写者写
 3. 在写的时候不能读
 4. 开始写的时候所有读要退出
---
两个进程：读进程，写进程
互斥关系：写-写，写-读
可以同时读

优化版：在count那里会并发吗导致两次P(rw)那么用一个信号量来防止一下做if判断时候的并发，也就是mutex
```c
semaphore rw = 1;        // 用于实现对共享文件的互斥访问
int count = 0;           // 记录当前有几个读进程在访问文件
semaphore mutex = 1;     // 用于保证对count变量的互斥访问 

writer () {
    while (1) {
        P(rw);           // 写之前"加锁"
        写文件...
        V(rw);           // 写完了"解锁"
    }
}

reader () {
    while (1) {
        P(mutex);        // 各读进程互斥访问count
        if (count == 0)  // 由第一个读进程负责
            P(rw);       // 读之前"加锁"
        count++;         // 访问文件的读进程数+1
        V(mutex);
        读文件...
        P(mutex);        // 各读进程互斥访问count
        count--;         // 访问文件的读进程数-1
        if (count == 0)  // 由最后一个读进程负责
            V(rw);       // 读完了"解锁"
        V(mutex);
    }
}
```

> **潜在问题**：只要有读进程还在读，写进程就要一直阻塞等待，可能**饿死**。此算法是读进程优先。

> **并发问题**：若两个读进程并发执行，count=0时两个进程都可能满足if条件，都会执行P(rw)，导致第二个读进程阻塞。解决方法：设置互斥信号量mutex保证对count的访问互斥，使检查和赋值一气呵成。

再次优化，再加一个信号量w

```c
semaphore rw = 1;        // 用于实现对共享文件的互斥访问
int count = 0;           // 记录当前有几个读进程在访问文件
semaphore mutex = 1;     // 用于保证对count变量的互斥访问
semaphore w = 1;         // 用于实现"写优先"

writer () {
    while (1) {
        P(w);            // 写者先抢 w，阻止后续读者
        P(rw);
        写文件...
        V(rw);
        V(w);
    }
}

reader () {
    while (1) {
        P(w);            // 读者也抢 w，但抢到后立即释放
        P(mutex);
        if (count == 0)
            P(rw);
        count++;
        V(mutex);
        V(w);            // 释放 w，让写者有机会抢
        读文件...
        P(mutex);
        count--;
        if (count == 0)
            V(rw);
        V(mutex);
    }
}
```

> **写优先原理**：增加信号量 `w=1`。写者和第一个读者都要 `P(w)`，但读者抢到 w 后立即 `V(w)` 释放，而写者持有 w 直到写完。一旦写者阻塞在 `P(rw)` 上，w 被它持有，后续读者全部阻塞在 `P(w)` → 写者优先获得 rw。

- 读者1→读者2：正常并发读
- 写者1→写者2：互斥写
- 写者1→读者1：写者先抢 w，读者阻塞
- 读者1→写者1→读者2：读者1释放 w 后写者抢到 w，读者2阻塞在 P(w)
- 写者1→读者1→写者2：写者1持 w 等 rw → 读者1阻塞在 P(w) → 写者1写完释放 rw 和 w → 写者2可抢



## 哲学家进餐
懒得写了

优化：
1. 同时允许四个人同时进餐
2. 对于奇数号的必须拿左边，偶数号必须先拿右边的
3. 同时拿起筷子的时候才可以进餐

```c
semaphore chopstick[5] = {1, 1, 1, 1, 1};
semaphore mutex = 1;      // 互斥地取筷子

Pi () {                   // i号哲学家的进程
    while (1) {
        P(mutex);
        P(chopstick[i]);           // 拿左
        P(chopstick[(i + 1) % 5]); // 拿右
        V(mutex);
        吃饭...
        V(chopstick[i]);           // 放左
        V(chopstick[(i + 1) % 5]); // 放右
        思考...
    }
}
```

> **死锁风险**：每个哲学家同时拿起左边筷子 → 所有人都在等右边的筷子 → 死锁。`mutex` 保证拿筷子过程互斥，避免了互相等待，但同一时刻只能一人拿筷子，效率低。
