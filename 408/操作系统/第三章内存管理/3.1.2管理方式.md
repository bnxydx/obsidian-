# 连续分配管理方式

## 单一连续分配

> 内存分为**系统区**（低地址，OS 专用）和**用户区**（仅一道用户程序独占）。
> 把内存全部分给一个进程
- **优点**：结构简单，**无外部碎片**，无须内存保护机制（内存始终只有一道程序）。
- **缺点**：仅适用于单用户、单任务 OS；存在**内部碎片**，内存利用率极低。


> **内部碎片**：分配给进程的**分区内部**未被利用的空间。分区大小 > 进程实际需求，多出的部分即内部碎片。

> **外部碎片**：分区**之间**的、因太小而无法分配给任何进程的**空闲块**。总空闲空间足够但分散不连续。
## 固定分区分配

> 最简单的多道程序存储管理方式
> 将用户空间划分为若干**大小固定**的分区，每个分区装入一道作业。

- **运行方式**：有空闲分区时，从外存后备作业队列选合适大小的作业装入，反复执行。
- **划分方法**：
  - **分区大小相等**：实现简单，但缺乏灵活性，程序大小与分区大小不匹配时浪费严重。
  - **分区大小不等**：可设置多个小分区、适量中等分区、少量大分区，适应不同大小的程序。
- **缺点**：存在**内部碎片**（程序小于分区时剩余空间浪费），且分区总数固定，限制了多道程序的道数。
- **无外部碎片**

### 基于顺序搜索的分配算法

#### 1. 首次适应（First Fit）

> 空闲分区按**地址递增**排序，从链首顺序查找第一个满足需求的空闲分区。

- 优先利用**低地址**分区，保留高地址大空闲区 → 有利于大作业装入。
- 缺点：低地址易积累**小碎片**；每次从链首查找，开销较大。

#### 2. 邻近适应（Next Fit / 循环首次适应）

> 在首次适应基础上改进：从**上次查找结束的位置**继续循环搜索，而非每次链首开始。

- 减少低地址区域重复扫描。
- 缺点：大空闲区分布不均，难以支持后续大作业装入，**整体性能通常不如首次适应**。

#### 3. 最佳适应（Best Fit）

> 空闲分区按**容量递增**排序，选择第一个满足需求的分区（即**最小的可用分区**）。

- 尽量保留大空闲区。
- 缺点：每次分配留下**极小剩余块**，随时间积累形成大量**外部碎片**，实际性能通常较差。

#### 4. 最坏适应（Worst Fit）

> 空闲分区按**容量递减**排序，选择**最大的空闲分区**进行分配。

- 分配后剩余空间较大，仍有被利用的可能，碎片产生较慢。
- 缺点：快速耗尽大空闲区，导致后续大作业难以装入。
### 基于索引搜索的分配算法

#### 1. 快速适应（Quick Fit）

> 按进程常用空间大小，预先将空闲分区划分为若干**固定尺寸类别**，用索引表管理各类别链表。

- **分配**：两步完成——①根据作业长度在索引表中定位能容纳它的**最小类别链表**；②从链表头部取下第一个空闲分区直接分配。
- **优点**：查找效率高（$O(1)$ 级索引定位），不会产生内存碎片。
- **缺点**：回收时难以有效合并分区，算法复杂、开销较大。

#### 2. 伙伴系统（Buddy System）

> 所有分区大小为 $2^k$（$k$ 为正整数）。分配时不断二分大块至目标大小，回收时与**伙伴分区**合并。

- **分配**：需求大小 $n$（$2^{i-1} < n \le 2^i$），先在 $2^i$ 链中查找；若无则向 $2^{i+1}, 2^{i+2}...$ 逐级上溯，找到后不断**二分**至所需大小，剩余块按大小加入对应空闲链。
- **伙伴**：大小相同、地址相邻（首尾相接）的两个分区互称伙伴。回收时若伙伴也为空闲则**合并**为 $2^{k+1}$ 分区，继续向上尝试合并直至无法合并。
- **优点**：分配与回收均有确定规则，碎片控制较好。

> **实例**：系统初始有 $128$MB 空闲空间。进程 $P_5$ 请求 $40$MB，$P_6$ 请求 $10$MB。

**分裂链**：$2 \to 4 \to 8 \to 16 \to 32 \to 64 \to 128$（每个大小可二分或合并）。

| 步骤  | 操作              | 说明                                                                                              |
| --- | --------------- | ----------------------------------------------------------------------------------------------- |
| ①   | $P_5$ 请求 $40$MB | $40$MB 向上取整至 $64$MB，在 $64$ 链无空闲 → 上溯至 $128$MB，二分得两个 $64$MB，一块分给 $P_5$                           |
| ②   | $P_6$ 请求 $10$MB | $10$MB 向上取整至 $16$MB，在 $16$ 链无空闲 → 上溯将剩余 $64$MB 不断二分：$64\to32+32\to32+16+16$，取一个 $16$MB 分给 $P_6$ |
| ③   | 最终状态            | 内存中留有 $32$MB、$16$MB 等多个空闲块，分布在对应大小链表中                                                           |
|     |                 |                                                                                                 |

> **回收时**：若释放的块与其伙伴均为空闲且同大小，则合并上溯直至无法合并。

#### 3. 哈希算法

> 以空闲分区**大小为关键字**建哈希函数，哈希表每个表项记录对应空闲分区链头指针。

- **分配**：根据所需分区大小，通过哈希函数计算表位置，快速定位对应空闲分区链。
- **优点**：查找效率高，$O(1)$ 级定位。
