# PCB进程控制块
为使每个参与并发执行的程序能独立运行，OS 为其配置一个专门的数据结构——**进程控制块（Process Control Block, PCB）**。操作系统通过 PCB 记录进程的基本信息和运行状态，从而实现对进程的控制与管理。

进程创建时，会创建一个PCB，常驻内存
进程终止时，PCB被回收
**是进程存在的唯一标志**
**通过PCB对并发执行的进程进行控制和管理的**
## 包含的内容

| 进程标识信息          | 进程调度信息       | 资源和内存信息     | 处理机状态信息         |
| --------------- | ------------ | ----------- | --------------- |
| **进程 ID (PID)** | **进程当前状态**   | 代码段指针       | **通用寄存器值**      |
| 父进程 ID (PPID)   | **进程优先级**    | 数据段指针       | **程序计数器值**      |
| 用户 ID (UID)     | **CPU 占用时间** | 堆栈段指针       | **程序状态字 (PSW)** |
|                 | 等待时间         | **文件描述符列表** | **栈指针 (SP)**    |
|                 | 进入内存时间       | I/O 资源清单    |                 |
|                 | 阻塞原因         |             |                 |
|                 | **CPU状态**    |             |                 |
# 程序段
可被CPU执行的代码
本身是静态的，可以被多个进程共享
# 数据段
每个进程的数据段是私有的

# 创建进程的流程
1. 分配唯一标识，申请PCB
2. 分配所需资源
3. 初始化PCB
4. 插入就绪队列


# 删除进程的流程
1. 根据被中值的标识符检索PCB，读取当前状态
2. 若运行，立即剥夺CPU
3. 释放所有资源，归还给操作系统
4. 递归终止子孙进程
5. 把PCB从队列中删除


## 数据分别存放位置
正文段：代码和赋值数据段，常量，全局赋值变量
数据堆段：动态分配的存储区
数据栈段：临时使用的变量（未赋值的局部变量，函数调用实参传递值）

# 进程的阻塞和唤醒
## 阻塞
当运行的进程因为外部事件无法继续，主动调用阻塞原语，**从运行态变成阻塞态**
等待的事件：
1. 请求系统资源不可用
2. 等待IO安慰昵称
3. 等待接收数据
4. 等待信号量

原语的流程：
1. 保存当前CPU现场到PCB（oc，psw）
2. 将PBC中的状态修改为阻塞，并插入等待事件的等待队列
3. 调用调度程序，让另一个程序运行
## 唤醒
当等待的事件完成后，中断处理程序或合作进程调用唤醒原语，激活进程
1. 找到在等待队列中的进程
2. 移除并修改位就绪
3. 插入就绪队列

**只要切换进程就需要内核的参与**
- **原因一：涉及敏感资源的修改（特权指令）**
    
    进程切换需要修改 CPU 内部的**页表基址寄存器（如 x86 的 CR3 寄存器）**，从而把内存映射切换到新进程的虚拟地址空间。修改页表、控制硬件寄存器属于**特权指令**，用户态（Ring 3）代码执行这些指令会直接触发保护异常。只有内核态（Ring 0）才有权限操作。
    
- **原因二：进程间是“平级”的，互不信任**
    
    $P_1$ 作为一个普通的用户进程，它根本没有资格、也不应该知道 $P_2$ 的 PCB 存放在哪，更没有权力去剥夺其他进程的 CPU 时间。只有作为“至高无上裁判”的**操作系统内核**，才能统筹全局进行调度。

当进程切换发生时，CPU 的状态一定会经历：**用户态 $\to$ 内核态 $\to$ 用户态** 的完整过程。
